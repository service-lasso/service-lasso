# Bootstrap Run Feedback

Run timestamp: `2026-04-05T11-02-18+10:00`

## Scope
This feedback reviews the bootstrap/setup process executed for this repository and is scrubbed for public issue filing.

## Summary
The bootstrap/setup process was easy to partially satisfy but hard to complete correctly because the required bar is split across multiple VibeGov sources with different levels of strictness. The governance files and pre-code artifacts were straightforward, but Git workflow, PR template, branch protection, board setup, and required reporting expectations were not consolidated tightly enough to prevent an incomplete bootstrap from feeling finished.

## What was underspecified or ambiguous
- The required pass gate appears inconsistent across the Bootstrap page, `agent.txt`, and `bootstrap.json`.
- It is unclear which source is canonical when they differ.
- The empty-repo path is underspecified when there is no validated product brief yet.
- It is unclear whether bootstrap must create `develop` immediately or only document the branch model.
- The exact required paths for the PR template and branch-protection checklist are not made explicit enough.
- GitHub board behavior is underspecified when auth works but board access, creation, or default project selection is unclear.

## What prerequisites or permissions were missing or should have been checked earlier
- `git` availability
- `gh` availability
- GitHub auth status
- repo admin / branch-protection access
- GitHub project board access
- ability to create or update workflow artifacts in `.github/`

## What artifacts or files should have been created earlier
- a durable bootstrap status artifact
- repo-local PR template
- branch-protection checklist
- explicit default issue-pickup flow
- a governance/bootstrap spec when product scope is still unknown

## What assumptions had to be invented
- product direction had to be inferred from the repository name in the absence of a validated product brief
- the first spec had to be framed as a likely initial slice rather than confirmed stakeholder intent
- repo-local backlog tracking had to stand in for GitHub issue/project normalization

## What should be mandatory next time
- one canonical bootstrap pass gate across all VibeGov sources
- an explicit empty-repo fallback path
- named capability checks with required outcome reporting
- tracked blockers when GitHub-dependent setup cannot be completed
- stable HTTPS bootstrap URLs without insecure redirects

## Exact wording or rules that would have made bootstrap smoother
> If the repository does not contain a validated product brief, do not infer product scope from the repository name or a placeholder README. Create `SPEC-001-bootstrap-governance-setup.md` as the first spec, mark project intent as provisional, and create a tracked blocker for missing domain intent.
>
> Bootstrap is incomplete unless one of the following is true for each GitHub-dependent step: `configured`, `blocked-with-tracked-issue`, or `not-applicable`. Report these states for: `git`, `gh`, GitHub auth, project-board access, branch-protection access, `develop` branch status, PR template, and branch-protection checklist.
>
> Use `https://vibegov.io/agent.txt` and `https://vibegov.io/bootstrap.json` as the canonical machine-readable bootstrap URLs. Do not rely on redirects from other hostnames.
>
> Create these workflow artifacts during bootstrap unless explicitly blocked: `develop` branch, repo-local pull request template, branch-protection checklist, and a documented default issue-pickup flow.
